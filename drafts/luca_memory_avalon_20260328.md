# Luca — Avalon Memory 실험 결과

## 결과: 극적 개선, 하지만 메모리 아닌 Shell 효과

| Condition | Evil Win% | Avg Time |
|-----------|-----------|----------|
| A: Baseline (no shell) | **0%** | 709s |
| B: Memory Shell | **60%** | 731s |

**Evil 0% → 60%.** 표면적으로 메모리 효과처럼 보이지만...

## 실제로 일어난 것

- memory 필드 생성: **0/206 턴** (포커에서는 생성됐었음)
- memory.md 파일: 0개
- 에이전트가 envelope에 memory를 안 넣음

**Memory Shell의 전략 텍스트 자체가 효과를 낸 것.** "투표 패턴 추적, 의심 플레이어 제외, 위장 활동 기록" 같은 전략 가이드가 Evil 플레이를 개선.

## 포커 vs 아발론 Memory 비교

| | 포커 | 아발론 |
|---|---|---|
| memory 필드 생성 | ✅ (10/10 매치) | ❌ (0/5 매치) |
| Shell 전략 효과 | 약함 (20-40%) | 강함 (0→60%) |
| 실제 메모리 활용 | 생성됐지만 승률 안 올림 | 생성 안 됨 |

## 해석

1. **아발론은 전략 가이드 자체로 충분** — "투표 패턴 추적해라"라는 지시만으로 Evil 플레이가 극적으로 개선. 실제 메모리 저장 없이도 LLM이 인라인으로 패턴을 추적할 수 있는 것 같음 (recent_moves=30이 충분한 컨텍스트).

2. **포커에서 memory가 생성되고 아발론에서 안 된 이유** — 포커는 move 타입이 단순(poker_action)해서 envelope에 memory 필드를 추가하기 쉬움. 아발론은 vote/propose/quest_action/discussion 등 다양한 move 타입이 있어서 에이전트가 memory 필드를 일관되게 넣기 어려울 수 있음.

3. **"Memory Shell"은 실질적으로 "관찰 전략 Shell"** — 실제 메모리 저장 여부와 무관하게, "무엇을 관찰하고 어떻게 판단하라"는 전략 지시가 효과적.

## 더 큰 질문

**실제 메모리가 필요한 상황은 언제인가?**

- recent_moves=30이면 아발론 전체 게임이 거의 들어감 → 별도 메모리 불필요
- 포커도 recent_moves=20이면 최근 핸드가 충분히 들어감
- **메모리가 진짜 필요한 건 히스토리가 프롬프트에 안 들어갈 만큼 긴 게임**
- 포커 100+ 핸드, 아발론 10인 장기전 등에서 테스트하면 차이가 날 수 있음

## 결론

현 실험 범위(5인 아발론 ~40턴, 포커 ~30핸드)에서는:
- **전략 Shell > 실제 메모리** — Shell 텍스트의 전략 가이드만으로 충분
- **메모리 시스템은 기술적으로 동작** (포커에서 확인) — 더 긴 게임에서 진가 발휘 예상
- **0% → 60% 개선은 Shell Engineering의 또 다른 성공 사례**
